## 6. Bug 修复记录

测试过程中一共发现并修复了 3 个较明显的问题，其他小问题（提示文案、前端空状态样式）在联调时顺手改掉，不单独记录。

### BUG-01 工单状态流转允许从“待接单”直接跳到“待确认”

| 项 | 内容 |
| --- | --- |
| 发现时间 | 2026-04-27 |
| 发现人 | 胡宇翔 |
| 关联用例 | `TC-W-06`、`TC-W-07` |
| 严重程度 | 高 |
| 状态 | 已修复 |

**问题现象**：

维修人员在工单状态还是 `PENDING_ACCEPT`（管理员刚分派过来，维修人员还没点“接单”）时，如果直接调用 `POST /api/orders/{id}/finish`，接口居然返回 200，工单状态被直接跳到 `PENDING_CONFIRM`。这样接单记录（`action_type=ACCEPT`）就丢了，维修记录看起来很奇怪。

**复现步骤**：

1. admin 分派工单 21 给 worker1，状态变为 `PENDING_ACCEPT`。
2. worker1 登录，但不点接单。
3. 用 Postman 直接调用：

   ```http
   POST /api/orders/21/finish
   Authorization: Bearer <worker1 token>
   {"resultDesc":"跳过接单试试"}
   ```

4. 返回 200，数据库 `status` 变为 `PENDING_CONFIRM`。

**根本原因**：

`RepairOrderService.finishOrder` 里只判断了“当前操作人是不是分派的维修人员”，没有判断当前工单状态是否为 `PROCESSING`，所以 `PENDING_ACCEPT` 也被放过了。

**修复方式**：

在 `finishOrder` 里加了一行状态校验：

```java
if (!OrderStatuses.PROCESSING.equals(order.getStatus())) {
    throw new BusinessException(ResultCode.CONFLICT, "只有处理中的工单才能完成维修");
}
```

同时顺手检查了 `acceptOrder`、`rejectByWorker`、`addRepairRecord`，把状态校验补齐。

**回归测试**：

- `TC-W-07` 用同样的请求重新跑一遍，返回 `409 只有处理中的工单才能完成维修`，工单状态保持 `PENDING_ACCEPT`。Pass。
- `TC-E-01` 端到端主流程重跑一次，状态流转正常。Pass。

**修复 commit**：`a20b3ce fix: 对齐维修人员接口校验与状态更新逻辑`

### BUG-02 JWT 过期后前端不退出，一直 401

| 项 | 内容 |
| --- | --- |
| 发现时间 | 2026-04-29 |
| 发现人 | 胡宇翔、尹添一 |
| 关联用例 | `TC-A-08` |
| 严重程度 | 中 |
| 状态 | 已修复 |

**问题现象**：

JWT 配置的过期时间是 2 小时。联调时把机器放了一晚上，第二天打开前端，点任何菜单都跳出一堆红色错误弹窗（`请求失败`），但页面不会自动跳到登录页，本地 token 也没清掉，用户只能手动刷新或清 localStorage。

**复现步骤**：

1. 登录 `student1`，记下当前 token。
2. 手动把 JWT 过期时间改成 30 秒（`application-dev.yml`），重启后端。
3. 等 1 分钟，点击任意菜单，例如“我的工单”。
4. 接口返回 401，页面弹出若干 `请求失败`，但没有自动跳登录。

**根本原因**：

前端 `utils/request.ts` 里的 axios 响应拦截器只对 `code !== 200` 打了个 `ElMessage.error(...)`，没处理 401 场景：既没清 token，也没跳登录，而且一个页面多个接口同时失败时会叠加好几个相同的错误弹窗。

**修复方式**：

在响应拦截器里增加 401 分支：

```ts
if (response.status === 401 || res.code === 401) {
  userStore.clearToken();
  ElMessage.error('登录已失效，请重新登录');
  router.replace('/login');
  return Promise.reject(new Error('unauthorized'));
}
```

并用 `ElMessage` 的 `grouping: true` 避免同一提示重复弹。

**回归测试**：

- `TC-A-08` 重跑：伪造 token 访问 `/api/orders/my`，前端自动清 token 并跳登录页，只弹一次“登录已失效”。Pass。
- 正常登录后长时间不操作再点击，也能正确跳回登录页。Pass。

**修复 commit**：尹添一负责，见前端 `utils/request.ts` 改动（本次提交已合入）。

### BUG-03 学生取消已驳回工单状态被错误置为 `CANCELLED`

| 项 | 内容 |
| --- | --- |
| 发现时间 | 2026-04-30 |
| 发现人 | 胡宇翔 |
| 关联用例 | `TC-S-15` |
| 严重程度 | 中 |
| 状态 | 已修复 |

**问题现象**：

工单被管理员驳回后状态为 `REJECTED`，前端已驳回页面本来不应该出现“取消报修”按钮。但用 Postman 直接打 `POST /api/orders/{id}/cancel` 时，接口仍然成功把工单状态改成了 `CANCELLED`，覆盖了原来的驳回原因。

**修复方式**：

`cancelOrder` 在查询工单时把状态条件收紧为 `PENDING_AUDIT`，其他状态一律抛 `CONFLICT`，和 `TC-S-15` 预期一致。修复后已通过回归。

**修复 commit**：随 `f1f5859 feat: 实现学生取消报修与评价接口` 后续联调修正一并提交。

### 9.3 端到端主流程实测（工单 id=11）

学生 student1 提交一条新工单，依次走完 6 步主流程：

| 步骤 | 接口 | HTTP / code | 工单状态变化 |
| --- | --- | --- | --- |
| 1. student1 创建 | `POST /api/orders` | 200 / 200 | → `PENDING_AUDIT` |
| 2. admin 审核通过 | `POST /api/orders/admin/11/approve` | 200 / 200 | → `PENDING_ASSIGN` |
| 3. admin 分派 worker1 | `POST /api/orders/admin/11/assign` | 200 / 200 | → `PENDING_ACCEPT` |
| 4. worker1 接单 | `POST /api/orders/11/accept` | 200 / 200 | → `PROCESSING` |
| 5. worker1 完成 | `POST /api/orders/11/finish` | 200 / 200 | → `PENDING_CONFIRM` |
| 6. student1 评价（5 分）| `POST /api/orders/11/confirm` | 200 / 200 | → `COMPLETED` |

最终详情接口 `GET /api/orders/11` 返回（节选）：

### 9.6 实测过程中发现并修正的小偏差

实测让我们也修复了文档与代码之间的几处不对齐，已在前面对应章节同步更新：

| 项 | 修正前 | 修正后（与代码一致） |
| --- | --- | --- |
| 分类列表路径 | `GET /api/categories` | `GET /api/category/list` |
| 审核请求体字段 | `{"remark":"..."}` | `{"adminRemark":"..."}` |
| 完成维修请求体字段 | `{"resultDesc":"..."}` | `{"actionDesc":"..."}` |
| 越权返回 msg | "无权限访问" | "仅管理员可以执行该操作" / "无权访问"（不同接口文案不同） |
| 工单号格式 | `WO202604280021`（12 位） | `WO2026050415005844B026BE`（带随机后缀，避免并发冲突） |

这些偏差已经在 §3、§5 各样例中修正。后续如再次实测出现新偏差，会按相同方式追加。
