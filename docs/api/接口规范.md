# 接口规范

本文档约定“校园宿舍报修与工单管理系统”前后端联调时的基础接口规则。后续新增接口默认遵循本规范，避免命名、返回结构和鉴权方式不一致。

## 1. 接口前缀

- 后端接口统一使用 `/api` 作为业务前缀。
- 本地开发时，前端通过 Vite 代理将 `/api` 请求转发到 `http://localhost:8080`。

示例：

- `POST /api/auth/login`
- `GET /api/user/me`
- `GET /api/orders`

## 2. 请求方式约定

- `GET`：查询数据
- `POST`：新增数据或执行会改变状态的动作
- `PUT`：整体更新
- `DELETE`：删除数据

涉及工单审核、分派、接单、确认完成等动作时，优先使用 `POST`，路径中体现动作语义。

示例：

- `POST /api/orders/{id}/approve`
- `POST /api/orders/{id}/assign`
- `POST /api/orders/{id}/accept`

## 3. 统一返回结构

所有接口统一返回 `code`、`msg`、`data` 三个字段。

成功示例：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "message": "pong"
  }
}
```

失败示例：

```json
{
  "code": 400,
  "msg": "请求参数错误",
  "data": null
}
```

字段说明：

- `code`：业务状态码
- `msg`：提示信息
- `data`：业务数据，失败时通常为 `null`

## 4. 常用响应码

| code | 含义 |
| --- | --- |
| `200` | 成功 |
| `400` | 请求参数错误 |
| `401` | 未登录或登录已失效 |
| `403` | 无权限访问 |
| `404` | 资源不存在 |
| `409` | 业务冲突 |
| `500` | 系统异常 |

## 5. 分页参数与返回格式

列表接口默认使用以下分页参数：

- `pageNum`：页码，从 `1` 开始
- `pageSize`：每页数量

分页返回放在 `data` 中，结构如下：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "records": [],
    "total": 0,
    "pageNum": 1,
    "pageSize": 10
  }
}
```

## 6. 时间字段格式

- 后端时间字段统一返回 `yyyy-MM-dd HH:mm:ss` 格式字符串。
- 数据库字段统一使用 `create_time`、`update_time`、`submit_time`、`finish_time` 这类蛇形命名。
- Java 字段统一使用驼峰命名，由 MyBatis-Plus 自动映射。

## 7. Token 传递方式

需要登录的接口统一通过请求头传 token：

```http
Authorization: Bearer <token>
```

约定：

- 登录成功后由后端返回 token
- 前端将 token 保存到本地状态和本地存储
- 未登录或 token 无效时，后端返回 `401`

## 8. 字段命名约定

- 数据库字段：下划线命名，如 `order_no`
- Java 字段：驼峰命名，如 `orderNo`
- JSON 字段：默认使用驼峰命名，如 `orderNo`

如果数据库状态值已经在数据库设计中固定，接口字段值应与数据库约定保持一致，不要另起一套名称。

## 9. 已确认的角色与状态取值

### 角色字段 `role`

- `STUDENT`：学生
- `ADMIN`：管理员
- `WORKER`：维修人员

### 用户状态字段 `status`

- `1`：正常
- `0`：禁用

### 分类状态字段 `status`

- `1`：启用
- `0`：停用

### 工单状态字段 `status`

- `PENDING_AUDIT`：待审核
- `PENDING_ASSIGN`：待分派
- `REJECTED`：已驳回
- `PENDING_ACCEPT`：待接单
- `PROCESSING`：处理中
- `PENDING_CONFIRM`：待确认
- `COMPLETED`：已完成
- `CLOSED`：已关闭

### 优先级字段 `priority`

- `LOW`：低
- `NORMAL`：普通
- `HIGH`：高
- `URGENT`：紧急

## 10. 登录接口返回结构

登录接口暂定为 `POST /api/auth/login`，成功时返回结构如下：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "token": "jwt-token",
    "userInfo": {
      "id": 1,
      "username": "admin",
      "realName": "系统管理员",
      "role": "ADMIN",
      "phone": "13800000001",
      "dormBuilding": null,
      "dormRoom": null,
      "status": 1
    }
  }
}
```

约定：

- `token` 为后续接口鉴权凭证
- `userInfo.role` 直接使用数据库中已确认的角色取值
- `userInfo.status` 直接使用数据库中已确认的状态取值

## 11. 空数据处理约定

- 查询成功但无数据时，列表返回空数组 `[]`
- 查询成功但无详情对象时，返回 `404`
- 删除、审核、分派、确认完成等动作成功时，`data` 可返回 `null`

## 12. 当前已确认的基础接口

- `GET /api/ping`：健康检查接口，当前用于验证统一返回格式
- `POST /api/auth/login`：后续登录接口
- `GET /api/user/me`：后续当前用户信息接口

后续新增接口时，请在此文档基础上继续补充。
