# 系统架构说明

## 一、总体结构

经典的"前后端分离 + 关系数据库 + 本地文件存储"三层结构，没有引入额外中间件，部署也简单。

```
┌─────────────────────────────────────────────────────┐
│  浏览器（学生 / 管理员 / 维修人员）                   │
└──────────────┬──────────────────────────────────────┘
               │  HTTP / JSON
               ▼
┌─────────────────────────────────────────────────────┐
│  Vue 3 前端    端口 5173                            │
│  - 路由：vue-router，按角色守卫                      │
│  - 状态：Pinia                                      │
│  - 请求：axios + 拦截器（自动带 token、401 退登）     │
│  - UI：Element Plus + ECharts                       │
└──────────────┬──────────────────────────────────────┘
               │  REST  /api/**   Bearer Token
               ▼
┌─────────────────────────────────────────────────────┐
│  Spring Boot 3.3 后端    端口 8080                  │
│  Controller → Service → Mapper                      │
│  公共组件：JWT、统一响应、全局异常、上传              │
└────────┬────────────────────┬───────────────────────┘
         │                    │
         ▼                    ▼
┌──────────────────┐    ┌──────────────────────┐
│  MySQL 8.0       │    │  本地文件系统          │
│  库 dorm_repair  │    │  backend/uploads/     │
│  utf8mb4         │    │  （图片+元数据存库）    │
└──────────────────┘    └──────────────────────┘
```

## 二、后端结构

### 2.1 技术选型

| 层 | 技术 | 备注 |

| 框架 | Spring Boot 3.3.5 | Java 21 |

### 2.2 包结构

按业务模块划分，每个模块都有自己的 controller / service / mapper / entity / vo。

```
com.dormrepair
├── DormRepairApplication        启动类
├── auth                         登录认证 + JWT 签发
├── user                         用户、角色、宿舍信息
├── category                     故障分类管理
├── order                        工单核心：审核、分派、接单、完成
├── record                       维修记录
├── evaluation                   学生评价
├── stats                        统计分析（5 个统计接口）
├── file                         文件上传 + 图片元数据
├── security                     JWT 拦截器、角色注解
├── config                       全局配置（CORS、MyBatis 等）
└── common                       统一响应体、异常处理、通用 VO
```

### 2.3 一次请求的处理链

以"学生提交报修"为例：

```
1. 前端 axios 把表单 + token 发到 POST /api/orders
2. 后端 JwtInterceptor 解析 token → 拿到 userId / role
3. RoleAware 注解检查角色：必须是 STUDENT
4. RepairOrderController 接收参数 → 转 Service
5. RepairOrderService.createOrder：
   - 校验分类是否启用
   - 生成 order_no（WO + 日期 + 序号）
   - 写 repair_order 表，初始状态 PENDING_AUDIT
6. 返回 { code: 200, msg: "ok", data: { id, orderNo } }
7. 全局异常处理器把任何抛出的 BusinessException 转成对应 HTTP 状态
```

### 2.4 公共组件

- **统一响应体**：所有接口走 `Result<T>`，固定 `{code, msg, data}`，前端拦截器只看这一层
- **全局异常**：业务异常 `BusinessException(code, msg)`，权限异常 `ForbiddenException`，参数异常用 `@Valid`
- **JWT 拦截器**：白名单（登录、健康检查）以外都校验 token，过期/无效都返回 401
- **角色控制**：自定义 `@RequireRole(ADMIN)` 注解，方法级生效
- **文件上传**：`POST /api/files/upload`，落 `backend/uploads/yyyy/MM/`，库里存 `file_storage` 表，业务表只存 url

### 3.3 路由守卫

进入需要登录的路由前会先看 Pinia 里有没有 token，没有就跳 `/login`。再按角色判断当前路由是不是允许：

- 学生只能进 `/home`、`/repair/*`
- 管理员只能进 `/home`、`/admin/*`
- 维修人员只能进 `/home`、`/work/*`

跨角色访问会被守卫直接踢回首页，后端那一层再兜一次 403。

### 3.4 axios 拦截器

请求拦截：自动带 `Authorization: Bearer <token>`。

响应拦截：

- HTTP 200 + code 200：正常往下走
- HTTP 200 + code 非 200：弹 `ElMessage.error(msg)`
- HTTP 401：清 token、跳 `/login`、提示"登录已失效"（多个并发只提示一次）
- HTTP 403：弹"没有权限"
- 其他：弹通用错误

## 四、部署架构

目前在本地跑，前后端各起一个进程：

```
开发模式
├── 前端：vite dev server     localhost:5173  proxy → /api → 8080
└── 后端：mvn spring-boot:run localhost:8080  连本地 MySQL
```

之后要打包部署的话也比较直接：

- 前端：`npm run build` → `dist/` 静态文件，丢 nginx 或者 Spring Boot 的 `static/`
- 后端：`mvn package` → `target/*.jar`，`java -jar` 跑
- 数据库：MySQL 单机
- 上传文件：和 jar 同目录的 `uploads/` 文件夹

## 五、为什么这么选

- **不上微服务**：项目体量小，5 张表 + 三类角色，单体足够，部署也方便
- **不引入 Redis**：JWT 是无状态的，会话不需要存储；统计接口数据量小，直接 SQL 查就够
- **不上 OSS**：图片本地存够用，加上 `file_storage` 表存元数据，后续要换 OSS 也只改文件服务一处
- **前后端分离**：方便分工，尹添一专心写 Vue，邓杰专心写后端，不用一起踩 Thymeleaf 这种坑
- **MyBatis-Plus 而不是 JPA**：复杂查询（统计、按状态聚合）用 SQL 直观，PageHelper 也省事
