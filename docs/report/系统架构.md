# 系统架构说明

## 一、总体结构

经典的"前后端分离 + 关系数据库"三层结构，没有引入额外中间件。上传的图片也以二进制形式存进 MySQL（`file_storage` 表），部署只需一个 JAR + 一台 MySQL，没有额外的文件目录要维护。

```
┌──────────────────────────────────────────────────────┐
│   浏览器（学生 / 管理员 / 维修人员）                   │
└────────────────────────┬─────────────────────────────┘
                         │  HTTP / JSON
                         ▼
┌──────────────────────────────────────────────────────┐
│   Vue 3 前端    端口 3000                            │
│   - 路由：vue-router，按角色守卫                      │
│   - 状态：Pinia                                       │
│   - 请求：axios + 拦截器（自动带 token、401 退登）     │
│   - UI：Element Plus + ECharts                        │
└────────────────────────┬─────────────────────────────┘
                         │  REST  /api/**   Bearer Token
                         ▼
┌──────────────────────────────────────────────────────┐
│   Spring Boot 3.3 后端    端口 8080                   │
│   Controller → Service → Mapper                       │
│   公共组件：JWT、统一响应、全局异常、上传服务           │
└────────────────────────┬─────────────────────────────┘
                         │  JDBC（含 LONGBLOB 读写）
                         ▼
┌──────────────────────────────────────────────────────┐
│   MySQL 8.0    库 dorm_repair    utf8mb4              │
│   ├─ 5 张业务表：user / repair_category /              │
│   │             repair_order / repair_record /         │
│   │             evaluation                             │
│   └─ 1 张文件表：file_storage  (LONGBLOB 存图片)        │
└──────────────────────────────────────────────────────┘
```

## 二、后端结构

### 2.1 技术选型

| 层 | 技术 | 备注 |
| --- | --- | --- |
| 框架 | Spring Boot 3.3.5 | Java 21 |
| Web | spring-web，REST 风格 | 全部 `/api` 前缀 |
| ORM | MyBatis-Plus 3.5 | 实体注解 + 自动分页 |
| 鉴权 | JWT（jjwt）+ 自定义拦截器 | token 有效期 2 小时 |
| 数据库 | MySQL 8.0 | utf8mb4，6 张表（5 业务表 + 1 文件存储表） |
| 构建 | Maven | 用 Spring Boot 父 POM |
| 工具 | Lombok 1.18.34 | 简化实体 |

### 2.2 包结构

按业务模块划分，核心模块有 controller / service / mapper / entity / vo，辅助模块只保留后两者。

```
com.dormrepair
├── DormRepairApplication        启动类
├── auth                         登录认证 + JWT 签发（controller/service/dto）
├── user                         用户、角色、宿舍信息
├── category                     故障分类管理
├── order                        工单核心：审核 / 分派 / 接单 / 记录 / 完成 / 评价  (21 个文件，最大模块)
├── record                       维修记录（entity + mapper，辅助 order 使用）
├── evaluation                   学生评价（entity + mapper，辅助 order 使用）
├── stats                        统计分析：5 个统计接口 + 对应 5 个 VO
│                                  Overview / CategoryDistribution /
│                                  WorkerLoad / DailyTrend / EvaluationStats
├── file                         文件上传 + 访问（LONGBLOB 存 MySQL）
├── security                     JWT 拦截器、角色注解 + AOP
├── config                       全局配置（CORS、MyBatis-Plus、Web 等）
└── common                       统一响应体 / 异常处理 / 通用常量 / 资源定位
```

> `record` 和 `evaluation` 这两个包只有实体和 Mapper，是因为业务逻辑（增加记录、提交评价）都是在工单主流程里调用的，放到 `order` 包里一起管事务更顺手。

### 2.3 一次请求的处理链

以"学生提交报修"为例：

```
1. 前端 axios 把表单 + token 发到 POST /api/orders
2. 后端 JwtInterceptor 解析 token → 拿到 userId / role
3. RoleAware 注解检查角色：必须是 STUDENT
4. RepairOrderController 接收参数 → 转 Service
5. RepairOrderService.createOrder：
   - 校验分类是否启用
   - 生成 order_no（WO + 日期 + 序号）
   - 写 repair_order 表，初始状态 PENDING_AUDIT
6. 返回 { code: 200, msg: "ok", data: { id, orderNo } }
7. 全局异常处理器把任何抛出的 BusinessException 转成对应 HTTP 状态
```

### 2.4 公共组件

- **统一响应体**：所有接口走 `Result<T>`，固定 `{code, msg, data}`，前端拦截器只看这一层
- **全局异常**：业务异常 `BusinessException(code, msg)`，权限异常 `ForbiddenException`，参数异常用 `@Valid`
- **JWT 拦截器**：白名单（登录、健康检查）以外都校验 token，过期/无效都返回 401
- **角色控制**：自定义 `@RequireRole(ADMIN)` 注解，方法级生效
- **文件上传**：`POST /api/files/upload` 上传、`GET /api/files/{id}` 读取。上传后二进制内容直接存进 `file_storage.file_data`（LONGBLOB），接口返回 `/api/files/{id}` 路径，业务表（`repair_order.image_url`、`repair_record.result_image`）仅以字符串引用，不建 FK。这样不依赖本地文件系统，打包部署也只需一个 JAR。

### 3.3 路由守卫

进入需要登录的路由前会先看 Pinia 里有没有 token，没有就跳 `/login`。再按角色判断当前路由是不是允许：

- 学生只能进 `/home`、`/repair/*`
- 管理员只能进 `/home`、`/admin/*`
- 维修人员只能进 `/home`、`/work/*`

跨角色访问会被守卫直接踢回首页，后端那一层再兜一次 403。

### 3.4 axios 拦截器

请求拦截：自动带 `Authorization: Bearer <token>`。

响应拦截：

- HTTP 200 + code 200：正常往下走
- HTTP 200 + code 非 200：弹 `ElMessage.error(msg)`
- HTTP 401：清 token、跳 `/login`、提示"登录已失效"（多个并发只提示一次）
- HTTP 403：弹"没有权限"
- 其他：弹通用错误

## 四、部署架构

目前在本地跑，前后端各起一个进程：

```
开发模式
├── 前端：vite dev server     localhost:3000  proxy → /api → 8080
└── 后端：mvn spring-boot:run localhost:8080  连本地 MySQL
```

之后要打包部署的话也比较直接：

- 前端：`npm run build` → `dist/` 静态文件，丢 nginx 或者 Spring Boot 的 `static/`
- 后端：`mvn package` → `target/*.jar`，`java -jar` 跑
- 数据库：MySQL 单机
- 上传图片：走 `file_storage` 表（LONGBLOB），不需要额外的文件目录。迁移、备份只需备份 MySQL。

## 五、为什么这么选

- **不上微服务**：项目体量小，6 张表 + 三类角色，单体足够，部署也方便
- **不引入 Redis**：JWT 是无状态的，会话不需要存储；统计接口数据量小，直接 SQL 查就够
- **不上 OSS 、图片也不走本地文件系统**：课程项目量级下直接存 MySQL `file_storage` 表 LONGBLOB 最简单。优点是部署不必多维护一份文件目录，备份迁移跟业务数据一体。文件服务被封装在 `file` 包里，后续要换 OSS 只需改 `FileService` 一处
- **前后端分离**：方便分工，尹添一专心写 Vue，邓杰专心写后端，不用一起踩 Thymeleaf 这种坑
- **MyBatis-Plus 而不是 JPA**：复杂查询（统计、按状态聚合）用 SQL 直观，PageHelper 也省事
