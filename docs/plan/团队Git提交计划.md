# 团队 Git 提交计划

本文档用于规划“校园宿舍报修与工单管理系统”开发过程中的团队 Git 提交节奏。它不是某一位成员的个人提交计划，而是全组围绕项目主流程形成的版本化成果计划。

项目开发采用“主流程优先、分工明确、持续留痕”的策略：先保证登录、报修、审核、分派、处理、评价这条核心业务链路完整可运行，再补充图片上传、统计分析、测试文档、演示数据和最终说明材料。

## 一、计划定位

### 1. 适用范围

本计划覆盖以下内容：

- 后端代码提交。
- 前端代码提交。
- 数据库脚本提交。
- 接口文档、测试记录、阶段报告、演示资料等过程文档提交。
- 前后端联调、Bug 修复、最终收尾提交。

### 2. 不是个人提交计划

本计划按功能和阶段组织，不按个人简单拆分。每一次提交都可以由一名成员主责，也可以由多名成员协作完成。

具体原则如下：

- 项目负责人负责把控主线、接口规范、合并节奏和最终可运行状态。
- 每位成员都应围绕自己的分工形成真实、连续、可追踪的 Git 提交。
- 不要求每个人都提交同样数量的代码，但要求每个人的提交内容能对应自己的项目职责。
- 第 13 次及之后的提交按单人职责拆分，每次提交只包含一个成员负责的功能或文档成果。
- 文档、测试、SQL、Postman 集合、演示资料同样属于有效 Git 成果。

## 四、提交原则

1. 每次 Git 提交只完成一个明确功能点，不把前端、后端、SQL、文档的无关修改混在一起。
2. 每次提交后项目应尽量保持可启动、可测试、可回退。
3. 每次提交都应有清晰验收标准，例如接口能访问、SQL 能执行、页面能打开、测试记录能复现。
4. 后端主线围绕核心流程推进：登录 -> 报修 -> 取消或审核 -> 分派 -> 处理 -> 确认完成与评价。
5. 前端主线围绕角色页面推进：登录页 -> 学生端 -> 管理端 -> 维修端 -> 统计页。
6. 数据库脚本、接口文档、测试记录、Postman 集合、会议纪要、阶段报告均纳入 Git 管理。
7. 每次提交如果影响启动方式、接口、数据库、页面进度或测试方式，应由本次提交负责人同步更新对应 Markdown 文档。
8. 每一步提交都应同步更新根目录 `README.md` 中的当前开发进度。
9. 不提交本地临时文件、IDE 缓存、上传目录、数据库密码、构建产物和无关截图。
10. 提交信息应能说明“谁完成了什么功能”，避免使用 `update`、`fix bug`、`修改` 这类模糊信息。
